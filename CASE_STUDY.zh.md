# 案例研究（中文底稿）— SERP Rank Tracker：可审计的每日 Google 排名报告

> 一个「记录失败而不是编造排名」的 Google 排名追踪器——面板上每个 live 实采数字都能审计回它来源的原始 API 响应。

**30 秒看懂**：打开 [live 面板](https://tommyz123.github.io/serp-rank-tracker/)，看顶部数据诚信面板，点任意行的验证抽屉看数字背后的证据，下载 CSV。代码：github.com/Tommyz123/serp-rank-tracker

*真实数据作品集项目：SerpAPI 每日实采自 2026-07-18 启动，live 历史逐日增厚。关键词是公开域名的公开搜索示例——无客户数据、无私有分析。*

## 它做什么
每日采集（SerpAPI 查询+域名匹配+证据入库 SQLite）；发布纯静态面板（诚信面板/趋势图/逐行验证抽屉/CSV 导出）；自动运行（每日 cron 采集→重建→发布，装好后低维护）。

## 对客户意味着什么
- 定时管道替代人工查排名——不用靠人记得去看
- 抓取失败永远不会被误认为排名下跌——失败入库并渲染成缺口，不被抹平进趋势线
- 每个报告名次带审计底单（可回溯到存档的原始响应）——agency 能向自己的客户证明报告为什么可信，而不是空口断言
- 报告干净导出（CSV 含完整溯源列），面板静态托管零成本
- 易定制：关键词/地区/域名/竞品/告警，都在同一个小 schema 上加

## 它解决的问题
排名追踪器容易无声失败：抓取超时但曲线依然平滑；演示数据与实测数据无声混合；"追踪 top 100"的口径超过解析器实际见过的范围。买 SEO 报告的人的核心问题是「这些数字是真的吗？」——本项目把诚实的答案做成**结构性的**：由 schema 和管道强制，不靠自觉。

## 诚信如何被强制
- 失败是一等公民行：3 次带标注重试，全败存为 failed（含错误+次数），缺口就是缺口
- 每个 live 数字有底单：原始 API 响应存档磁盘、路径与解析结果同存，UI 逐行验证抽屉（seeded 演示行标注为合成、不参与 live 声明）
- 合成历史构造上就是独立层：layer='seeded' 由 SQLite CHECK 约束强制，空心标记+徽章+CSV layer 列；seeded 里故意含一次失败 run——demo 连自己的失败处理都展示
- 深度实录不空口：每次 run 存 result_depth 供审计；demo 追踪并标注 Google 第一页

## 刻意做小
静态面板（免费托管/秒开/访客不烧配额）；纯标准库（没东西可坏、交接零门槛）；配额敏感（一次搜索覆盖该词全部域名，成本随关键词数走、与盯几个域名无关）。

## 上线前验证（五项）
①存储↔存档逐条核对 ②CSV 导出完整（含 failed/seeded 行及 layer 列）③分层纯净端到端 ④浏览器实测 JS 零报错、抽屉与导出全可用 ⑤故障路径实测（真失败被如实记录为失败）。建造前另针对买家可能的质疑做了压力测试——所以诚信面板顶在首屏、范围收在访客 30 秒内可验证的东西。

## 客户工作的自然延伸
掉名告警邮件 · 竞品列 · 多地区追踪 · 更深分页 · 定时 PDF/邮件报告——都是同一 runs/checks schema 上的直接加法。

### 技术栈
Python 3.10+（纯标准库）· SQLite · SerpAPI · GitHub Pages · cron
